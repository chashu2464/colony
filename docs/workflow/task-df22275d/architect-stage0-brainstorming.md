# Proposal B Stage 0 Brainstorming（Architect）

- Task ID: `df22275d`
- Date: `2026-03-31`
- 任务类型分类：**新功能**（流程编排能力增强，涉及阶段流转与 agent 唤醒链路）
- 选型工作流：**dev-workflow**（跨模块、多角色、含架构与测试门禁）

## 1) 问题定义
当前我们需要把“阶段流转 -> 目标执行者唤醒 -> 执行结果回写”的链路做成更确定、更可审计的机制，减少人工 @ 驱动与隐式状态推进带来的不确定性。

## 2) 借鉴目标（来自 clowder-ai 对比）
1. 显式路由目标（下一阶段责任人明确可计算）
2. 队列化执行（避免并发踩踏，保证可重放）
3. 流程推进与执行触发解耦（先落状态，再触发执行）
4. 失败闭环（失败有 machine-readable reason，可重试/回滚）

## 3) 本项目落地边界（Phase 1）
- 范围内：`dev-workflow` 的阶段推进触发链路、审计字段、可观测事件、失败闭环。
- 范围外：`quick-task` 联动（放到 Phase 2）。

## 4) 基础设施可行性检查
- 现状：`dev-workflow` 已在 `next/backtrack` 后调用 `notify_server`，具备“阶段变化后通知下游”的基础触发能力。
- 结论：无需引入新调度系统即可先实现 Phase 1；通过扩展通知载荷和路由规则即可支撑“确定性唤醒”。
- 理由（一句话）：复用现有通知链路，最小侵入、上线风险最低。

## 5) 风险清单（必须管控）
1. 重复唤醒：同一阶段重复触发导致多 agent 重入。
2. 顺序错乱：状态未持久化完成即触发执行。
3. 黑盒失败：只知道失败、不知道失败原因。
4. 角色漂移：阶段与角色映射不一致导致执行落空。

## 6) Stage 1 输出目标
在 IR 中冻结：
- 流程流转契约
- 唤醒契约
- 审计/可观测字段
- 失败闭环与验收标准
