# P3阶段需求文档

## 任务概述
**任务ID**: 40395d8a
**任务名称**: P3阶段：单一数据源重构与智能上下文压缩
**优先级**: P3（长期优化）
**预计工期**: 2-3个月

## 背景与动机

### 当前问题
1. **角色映射逻辑重复**：`get_next_actor()` 和 `getStageGuidanceForAgent()` 存在重复的角色映射代码
2. **配置分散**：CLI并发控制等配置硬编码在代码中
3. **上下文容量限制**：当前只保留20条消息，长对话场景下上下文丢失严重
4. **Token成本高**：完整历史消息占用大量token

### 预期收益
- 统一数据源，降低维护成本
- 配置外部化，提升灵活性
- 智能压缩上下文，在有限token内保留更多历史
- 提升长对话场景下的Agent理解能力

## 功能需求

### 方向1：单一数据源重构

#### FR1.1 Markdown解析器
**描述**: 实现零依赖的Markdown表格解析器，从SKILL.md提取阶段-角色映射表

**输入**: SKILL.md文件路径
**输出**: 结构化的阶段-角色映射数据

**数据结构**:
```typescript
interface StageRoleMapping {
  stage: number;
  stageName: string;
  primaryRole: string;
  collaborators: string[];
  guidance: string;
}
```

**性能要求**: 解析耗时 < 5ms

#### FR1.2 统一角色映射逻辑
**描述**: 消除 `get_next_actor()` 和 `getStageGuidanceForAgent()` 的重复代码

**改动范围**:
- `skills/dev-workflow/scripts/handler.sh` - 修改 `get_next_actor()`
- `src/memory/ContextAssembler.ts` - 修改 `getStageGuidanceForAgent()`

**实现方式**: 两者都从解析后的SKILL.md数据中读取

#### FR1.3 环境变量配置加载
**描述**: 支持从环境变量加载配置参数

**配置项**:
- `COLONY_MAX_CLI_CONCURRENCY`: CLI最大并发数（默认: 3）
- 未来可扩展其他配置项

**加载时机**: 系统启动时

#### FR1.4 数据验证与降级
**描述**: 完善的错误处理和降级策略

**验证规则**:
- 阶段编号必须连续（0-9）
- 每个阶段必须有主要负责人
- 角色名称必须在已知角色列表中

**降级策略**:
- 解析失败时使用硬编码的默认映射表
- 记录警告日志但不中断服务

### 方向2：智能上下文压缩

#### FR2.1 上下文压缩算法设计
**描述**: 设计分层压缩策略

**压缩策略**:
1. **最近N条消息**: 完整保留（N=10）
2. **中期消息**: 使用LLM压缩摘要（10-50条前）
3. **早期消息**: 仅保留关键决策点

**压缩触发条件**: 消息数 > 20条

#### FR2.2 LLM压缩实现
**描述**: 使用LLM对中期消息进行智能压缩

**压缩Prompt模板**:
```
请将以下对话压缩为简洁摘要，保留：
1. 关键决策和结论
2. 重要的技术细节
3. 未解决的问题

对话内容：
[消息列表]
```

**压缩比目标**: 原始token的30-40%

#### FR2.3 压缩缓存机制
**描述**: 缓存已压缩的消息段，避免重复压缩

**缓存策略**:
- 以消息范围为key（如"msg_10_to_30"）
- 缓存过期时间：会话结束时清理
- 内存限制：单会话最多缓存5个压缩段

## 非功能需求

### NFR1 性能
- Markdown解析: < 5ms
- 上下文压缩: < 2s（异步执行，不阻塞主流程）
- 内存占用: 压缩后减少60%以上

### NFR2 可维护性
- 零外部依赖（Markdown解析使用原生正则）
- 完善的单元测试覆盖率 > 80%
- 清晰的错误日志和降级提示

### NFR3 兼容性
- 向后兼容：解析失败时使用现有硬编码逻辑
- 不影响现有功能：渐进式迁移

## 验收标准

### AC1 单一数据源重构
- [ ] Markdown解析器能正确解析SKILL.md中的阶段-角色映射表
- [ ] `get_next_actor()` 和 `getStageGuidanceForAgent()` 都从解析结果读取数据
- [ ] 环境变量 `COLONY_MAX_CLI_CONCURRENCY` 生效
- [ ] 解析失败时能正确降级到默认配置
- [ ] 所有现有工作流测试通过

### AC2 智能上下文压缩
- [ ] 消息数 > 20时自动触发压缩
- [ ] 压缩后的上下文能正确传递给Agent
- [ ] Agent能理解压缩后的上下文（通过人工测试验证）
- [ ] 压缩比达到60%以上
- [ ] 压缩过程不阻塞消息发送

## 实施计划

### 阶段划分
1. **Stage 1-2**: 需求分析与架构设计（当前）
2. **Stage 3-5**: 设计评审与测试用例编写
3. **Stage 6**: 开发实施
   - 先实现方向1（单一数据源重构）
   - 再实现方向2（智能上下文压缩）
4. **Stage 7-8**: 集成测试与上线评审

### 风险评估
| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| Markdown解析复杂度超预期 | 中 | 低 | 使用成熟的正则表达式模式 |
| LLM压缩质量不稳定 | 高 | 中 | 设计详细的压缩prompt，多轮测试优化 |
| 性能影响主流程 | 高 | 低 | 异步执行压缩，设置超时保护 |
| 向后兼容性问题 | 中 | 低 | 完善的降级机制和测试覆盖 |
